# Monitoring Module

## Overview

The `monitoring` module provisions the security and compliance notification layer for a `tf-secure-baseline` workload environment.

This module creates encrypted SNS and SQS resources, EventBridge alert targets, CloudWatch metric filters, CloudWatch alarms, and notification delivery paths used by the rest of the baseline.

It is responsible for alert routing, durable notification buffering, and human-readable SecOps notifications. It does not create every detection source by itself; instead, it connects CloudTrail, CloudWatch, EventBridge, Security Hub, AWS Config, Lambda publishers, and security automation workflows to notification targets.

This module currently includes:

- Compliance SNS topic
- Compliance SQS queue
- Security notifications SNS topic
- Security notifications SQS queue
- Security notifications SQS DLQ
- Security notifications EventBridge DLQ
- SecOps email subscriptions
- SNS topic policies for AWS service, EventBridge, CloudWatch, and Lambda publishing
- SQS queue policies for SNS and EventBridge delivery paths
- EventBridge target for Security Hub high and critical findings
- EventBridge detection for break-glass role usage
- EventBridge target support for tamper-detection alerts created outside this module
- CloudWatch Log Metric Filters for CloudTrail-based detections
- CloudWatch Alarms routed to security notifications

---

## Purpose

The purpose of this module is to centralize operational, security, and compliance notifications.

It supports:

- Security alert delivery to SecOps email recipients
- Durable security notification buffering through SQS
- Durable compliance notification buffering through SQS
- EventBridge target failure retention through a shared EventBridge DLQ
- Alerting on high and critical Security Hub findings
- Alerting on break-glass role usage
- Alerting on tamper-detection events generated by the security module
- Alerting on root account activity
- Alerting on unauthorized API calls
- Alerting on CloudTrail tampering
- Alerting on selected IAM policy and trust policy changes
- Queue-based evidence retention for notification and failure paths

This module acts as the alert routing layer for the baseline.

---

## Design Summary

The module separates notification delivery into three related paths:

| Path | Purpose | Durable Queue / DLQ |
|---|---|---|
| Compliance notifications | AWS Config and compliance event delivery | Compliance SQS queue |
| Security notifications | SecOps alert fanout and future downstream consumers | Security notifications SQS queue and SQS redrive DLQ |
| EventBridge security notification failures | Failed EventBridge delivery to the security SNS topic | Shared security notifications EventBridge DLQ |

The security notification path is intentionally split into two DLQ concepts:

| Resource | What it protects |
|---|---|
| `security-notifications-dlq` | Failed or repeatedly unprocessed messages from the security notifications SQS queue |
| `security-notifications-eventbridge-dlq` | Failed EventBridge target deliveries to the security notifications SNS topic |

These queues cover different failure points and should not be collapsed into a single queue.

## Resources Created

### Compliance SNS Topic

Creates a compliance notification SNS topic:

```hcl
resource "aws_sns_topic" "compliance"
```

Topic name format:

```text
<name_prefix>-compliance-notifications
```

The topic is encrypted with the logs CMK:

```hcl
kms_master_key_id = var.logs_cmk_arn
```

This topic is intended for AWS Config and compliance-related notifications.

---

### Compliance SNS Topic Policy

Creates a topic policy for the compliance SNS topic:

```hcl
resource "aws_sns_topic_policy" "compliance"
```

The policy allows:

- Account root administration
- AWS Config to publish compliance notifications

The AWS Config service principal is:

```text
config.amazonaws.com
```

---

### Compliance SQS Queue

Creates a compliance SQS queue:

```hcl
resource "aws_sqs_queue" "compliance"
```

Queue name format:

```text
<name_prefix>-compliance-queue
```

The queue is encrypted with the logs CMK:

```hcl
kms_master_key_id = var.logs_cmk_arn
```

The queue uses 14-day message retention:

```hcl
message_retention_seconds = 1209600
```

This queue receives messages from the compliance SNS topic.

The compliance queue is a durable notification subscriber intended for buffering, replay, evidence collection, troubleshooting, and future downstream consumers. It is not required to have an active automated consumer in the baseline.

---

### Compliance SNS Subscription

Subscribes the compliance SQS queue to the compliance SNS topic:

```hcl
resource "aws_sns_topic_subscription" "compliance"
```

Subscription protocol:

```text
sqs
```

This allows compliance notifications published to SNS to be delivered into the SQS queue.

---

### Compliance SQS Queue Policy

Creates an SQS queue policy that allows the compliance SNS topic to send messages to the compliance queue:

```hcl
data "aws_iam_policy_document" "compliance_queue_policy"
resource "aws_sqs_queue_policy" "compliance"
```

The policy allows:

```text
sqs:SendMessage
```

Only from the compliance SNS topic ARN.

This prevents unrelated SNS topics or principals from writing to the compliance queue.

---

### Security Notifications SNS Topic

Creates the main security notification SNS topic:

```hcl
resource "aws_sns_topic" "secops"
```

Topic name format:

```text
<name_prefix>-security-notifications
```

The topic is encrypted with the logs CMK:

```hcl
kms_master_key_id = var.logs_cmk_arn
```

This topic receives security notifications from:

- CloudWatch alarms
- EventBridge rules
- Security Hub event routing
- Tamper detection
- Break-glass detection
- EC2 Isolation Lambda
- EC2 Rollback Lambda
- IP Enrichment Lambda

The topic fans out security notifications to:

- Confirmed SecOps email subscriptions
- The security notifications SQS queue

---

### Security Notifications SNS Topic Policy

Creates a topic policy for the security notifications SNS topic:

```hcl
resource "aws_sns_topic_policy" "secops"
```

The policy allows publishing from:

| Principal / Source | Purpose |
|---|---|
| Account root | Topic administration |
| `cloudwatch.amazonaws.com` | CloudWatch alarm notifications |
| `events.amazonaws.com` | EventBridge rule notifications |
| IP Enrichment Lambda role | Threat intel enrichment alerts |
| EC2 Isolation Lambda role | Instance isolation alerts |
| EC2 Rollback Lambda role | Instance rollback alerts |
| Break-glass EventBridge rule | Break-glass role usage alerts |
| Tamper detection EventBridge rule | Security service tampering alerts |
| Security Hub high/critical rule | High and critical Security Hub finding alerts |

EventBridge permissions are scoped with source account and source ARN conditions where applicable.

---

### SecOps Email Subscriptions

Creates email subscriptions for each address in the `secops_emails` variable:

```hcl
resource "aws_sns_topic_subscription" "secops"
```

The module uses:

```hcl
for_each = toset(var.secops_emails)
```

Each email address receives a subscription to the security notifications SNS topic.

Important:

Email subscriptions require confirmation by the recipient before notifications are delivered.

### Security Notifications SQS Queue

Creates a security notifications SQS queue:

```hcl
resource "aws_sqs_queue" "security_notifications"
```

Queue name format:

```text
<name_prefix>-security-notifications-queue
```

The queue is encrypted with the logs CMK:

```hcl
kms_master_key_id = var.logs_cmk_arn
```

The queue uses 14-day message retention:

```hcl
message_retention_seconds = 1209600
```

This queue receives messages from the security notifications SNS topic.

The queue provides a durable machine-readable copy of security notifications for troubleshooting, replay, evidence collection, and future downstream integrations such as SIEM ingestion, ticketing, or client-owned consumers.

The baseline does not require this queue to have an active consumer. If no consumer is configured, visible messages may accumulate until they expire or are manually drained.

---

### Security Notifications SQS DLQ

Creates a DLQ for the security notifications SQS queue:

```hcl
resource "aws_sqs_queue" "security_notifications_dlq"
```

Queue name format:

```text
<name_prefix>-security-notifications-dlq
```

The queue is encrypted with the logs CMK:

```hcl
kms_master_key_id = var.logs_cmk_arn
```

The security notifications queue redrives messages to this DLQ after repeated failed receive attempts:

```hcl
redrive_policy = jsonencode({
  deadLetterTargetArn = aws_sqs_queue.security_notifications_dlq.arn
  maxReceiveCount     = 5
})
```

This DLQ protects the SNS-to-SQS notification consumption path.

It does not protect EventBridge target delivery failures. EventBridge target delivery failures are handled by the separate security notifications EventBridge DLQ.

---

### Security Notifications DLQ Alarm

Creates a CloudWatch alarm for visible messages in the security notifications SQS DLQ:

```hcl
resource "aws_cloudwatch_metric_alarm" "security_notifications_dlq_visible_messages"
```

Alarm name format:

```text
<name_prefix>-security-notifications-dlq-visible-messages
```

The alarm watches:

```text
AWS/SQS ApproximateNumberOfMessagesVisible
```

The alarm triggers when the DLQ has visible messages and routes the alarm to the security notifications SNS topic.

A visible message in this DLQ should be reviewed by SecOps or an authorized operator.

---

### Security Notifications SNS to SQS Subscription

Subscribes the security notifications SQS queue to the security notifications SNS topic:

```hcl
resource "aws_sns_topic_subscription" "security_notifications_sqs"
```

Subscription protocol:

```text
sqs
```

Raw message delivery is enabled:

```hcl
raw_message_delivery = true
```

The subscription depends on the SQS queue policy so SNS can publish to the queue:

```hcl
depends_on = [
  aws_sqs_queue_policy.security_notifications
]
```


The policy allows:

```text

```

Only from the security notifications SNS topic ARN.

### Security Notifications EventBridge DLQ

Creates a shared EventBridge target DLQ for failed EventBridge deliveries to the security notifications SNS topic:

```hcl
resource "aws_sqs_queue" "security_notifications_eventbridge_dlq"
```

Queue name format:

```text
<name_prefix>-security-notifications-eventbridge-dlq
```

The queue is encrypted with the logs CMK:

```hcl
kms_master_key_id = var.logs_cmk_arn
```

The queue uses 14-day message retention:

```hcl
message_retention_seconds = 1209600
```

This DLQ is used by EventBridge targets that send security alerts to the security notifications SNS topic.

Current protected EventBridge-to-SNS targets include:

| EventBridge rule | Target | DLQ |
|---|---|---|
| Security Hub high/critical rule | Security notifications SNS topic | Security notifications EventBridge DLQ |
| Break-glass role assumed rule | Security notifications SNS topic | Security notifications EventBridge DLQ |
| Tamper detection rule | Security notifications SNS topic | Security notifications EventBridge DLQ |

This queue is specifically for EventBridge target delivery failures. It is separate from the security notifications SQS queue DLQ.

---

### Security Notifications EventBridge DLQ Policy

Creates an SQS queue policy that allows EventBridge to send failed target delivery events to the security notifications EventBridge DLQ:

```hcl
data "aws_iam_policy_document" "security_notifications_eventbridge_dlq"
resource "aws_sqs_queue_policy" "security_notifications_eventbridge_dlq"
```

The policy allows:

```text
sqs:SendMessage
```

from the EventBridge service principal:

```text
events.amazonaws.com
```

The policy is scoped to the EventBridge source rule ARNs for:

- Break-glass role assumption detection
- Security Hub high/critical findings
- Tamper detection

---

### Security Hub High/Critical EventBridge Target

Creates an EventBridge target that sends high and critical Security Hub findings to the security notifications SNS topic:

```hcl
resource "aws_cloudwatch_event_target" "securityhub_high_critical"
```

Important:

The EventBridge rule itself is created outside this module, in the automation layer.

This module receives the rule name and rule ARN through variables:

```hcl
var.securityhub_high_critical_rule_name
var.securityhub_high_critical_rule_arn
```

Target ID:

```text
sec-hub-to-secops-sns
```

The target uses:

```hcl
dead_letter_config {
  arn = aws_sqs_queue.security_notifications_eventbridge_dlq.arn
}

retry_policy {
  maximum_event_age_in_seconds = 3600
  maximum_retry_attempts       = 3
}
```

The target uses an input transformer to format Security Hub findings into readable alert messages.

The alert includes:

- Severity
- Product name
- Finding title
- Time
- Account
- Region
- Finding ID
- Workflow status
- Record state
- Resource type
- Resource ID

---

### Break-Glass Role Assumption Detection Rule

Creates an EventBridge rule that detects when the break-glass admin role is assumed:

```hcl
resource "aws_cloudwatch_event_rule" "break_glass_assumed"
```

Rule name format:

```text
<name_prefix>-break-glass-admin-assumed
```

The rule matches CloudTrail events for:

```text
eventSource = sts.amazonaws.com
eventName   = AssumeRole
roleArn     = var.break_glass_admin_role_arn
```

This is intended to alert whenever emergency administrative access is used.

---

### Break-Glass EventBridge Target

Creates an EventBridge target that sends break-glass role assumption alerts to the security notifications SNS topic:

```hcl
resource "aws_cloudwatch_event_target" "break_glass_assumed_to_sns"
```

Target ID:

```text
break-glass-to-secops-sns
```

The target uses:

```hcl
dead_letter_config {
  arn = aws_sqs_queue.security_notifications_eventbridge_dlq.arn
}

retry_policy {
  maximum_event_age_in_seconds = 3600
  maximum_retry_attempts       = 3

```

The alert includes:

- Time
- Account
- Region
- Caller ARN

- Session name

Break-glass role usage should be treated as critical and reviewed immediately.

---

### Tamper Detection EventBridge Target Support

The tamper detection EventBridge rule is created outside this module and passed in through:

```hcl
var.tamper_detection_rule_arn
```

This module allows the tamper detection rule to publish alerts to the security notifications SNS topic through the SNS topic policy.

The security notifications EventBridge DLQ policy also allows the tamper detection EventBridge rule to send failed target delivery events to:

```text
<name_prefix>-security-notifications-eventbridge-dlq
```

The actual tamper detection EventBridge target may be created outside this module, but it should use the same security notifications SNS topic, shared EventBridge DLQ, and retry policy.

## CloudWatch Log Metric Filters and Alarms

The module creates several CloudWatch Log Metric Filters based on the CloudTrail log group.

The CloudTrail log group name is passed in through:

```hcl
var.cloudtrail_logs_group_name
```

Each metric filter creates a custom metric in the namespace:

```hcl
var.name_prefix
```

Each related alarm sends notifications to the security notifications SNS topic.

---

### Root Activity Detection

Creates a metric filter for root user activity:

```hcl
resource "aws_cloudwatch_log_metric_filter" "root_activity"
resource "aws_cloudwatch_metric_alarm" "root_activity"
```

Filter pattern:

```text
{$.userIdentity.type = "Root"}
```

Metric name:

```text
RootActivityCount
```

Alarm name format:

```text
<name_prefix>-Root-User-Activity
```

Expected behavior:

- Any root user activity creates a metric value.
- The alarm triggers when root activity count is greater than or equal to 1 within a 5-minute period.

Root account usage should be rare and should be reviewed.

---

### Unauthorized API Call Detection

Creates a metric filter for unauthorized API activity:

```hcl
resource "aws_cloudwatch_log_metric_filter" "unauthorized_api_calls"
resource "aws_cloudwatch_metric_alarm" "unauthorized_api_calls"
```

Filter pattern detects:

```text
UnauthorizedOperation
AccessDenied
```

Metric name:

```text
UnauthorizedAPICallCount
```

Alarm name format:

```text
<name_prefix>-Unauthorized_API_Calls
```

Expected behavior:

- Any denied API call creates a metric value.
- The alarm triggers when unauthorized API call count is greater than or equal to 1 within a 5-minute period.

This is useful for detecting misconfigured roles, suspicious enumeration, failed privilege attempts, or normal least-privilege tuning events.

---

### CloudTrail Disabled Detection

Creates a metric filter for CloudTrail tampering events:

```hcl
resource "aws_cloudwatch_log_metric_filter" "cloudtrail_disabled"
resource "aws_cloudwatch_metric_alarm" "cloudtrail_disabled"
```

Filter pattern detects:

```text
StopLogging
DeleteTrail
```

Metric name:

```text
CloudTrailDisabled
```

Alarm name format:

```text
<name_prefix>-CloudTrailDisabled
```

Expected behavior:

- Attempts to stop or delete CloudTrail create a metric value.
- The alarm triggers when the count is greater than or equal to 1 within a 5-minute period.

This is a high-priority security alert.

---

### IAM Policy Change Detection

Creates a metric filter for selected IAM policy and role trust policy changes:

```hcl
resource "aws_cloudwatch_log_metric_filter" "iam_policy_changes"
resource "aws_cloudwatch_metric_alarm" "iam_changes"
```

Current filter detects:

```text
CreatePolicy
PutRolePolicy
AttachRolePolicy
DeletePolicy
DetachRolePolicy
UpdateAssumeRolePolicy
```

Metric name:

```text
IamPolicyChanges
```

Alarm name format:

```text
<name_prefix>-IamPolicyChanges
```

Expected behavior:

- Selected IAM policy changes create a metric value.
- Selected IAM policy detach/delete events create a metric value.
- Role trust policy updates create a metric value.
- The alarm triggers when IAM policy change count is greater than or equal to 1 within a 5-minute period.

Security relevance:

- `CreatePolicy`, `PutRolePolicy`, and `AttachRolePolicy` may indicate privilege creation, privilege expansion, or policy attachment activity.
- `DeletePolicy` and `DetachRolePolicy` may indicate attempted permission removal, cleanup, or defense evasion.
- `UpdateAssumeRolePolicy` is especially important because it changes who can assume a role, which can create privilege escalation or persistence risk.

## Alert Routing

### Compliance Notification Path

Compliance notifications follow this path:

```text
AWS Config
    |
    v
Compliance SNS Topic
    |
    v
Compliance SQS Queue
```

This allows compliance events to be queued for later processing, inspection, troubleshooting, replay, or integration with another system.

---

### Security Notification Path

Security notifications follow this path:

```text
CloudWatch Alarms / EventBridge Rules / Lambda Functions
    |
    v
Security Notifications SNS Topic
    |
    +--> SecOps Email Subscriptions
    |
    +--> Security Notifications SQS Queue
             |
             v
         Security Notifications SQS DLQ
         after repeated failed receive attempts
```

This supports direct human notification and durable machine-readable notification retention.

---

### EventBridge Security Notification Failure Path

EventBridge targets that send security alerts to the security notifications SNS topic use the shared EventBridge DLQ:

```text
EventBridge Rule
    |
    v
Security Notifications SNS Topic target
    |
    | delivery failure after retry policy
    v
Security Notifications EventBridge DLQ
```

The protected EventBridge-to-SNS paths are:

- Security Hub high/critical findings
- Break-glass role assumption alerts
- Tamper detection alerts

Each protected EventBridge target uses:

```text
maximum_retry_attempts       = 3
maximum_event_age_in_seconds = 3600
```

---

### Security Hub Notification Path

Security Hub high and critical findings follow this path:

```text
Security Hub Finding
    |
    v
EventBridge Rule from automation module
    |
    v
EventBridge Target from monitoring module
    |
    +--> Security Notifications SNS Topic
    |       |
    |       +--> SecOps Email Subscriptions
    |       +--> Security Notifications SQS Queue
    |
    +--> Security Notifications EventBridge DLQ
            if EventBridge cannot deliver to SNS
```

The rule is created in the automation module.

The SNS target and EventBridge target DLQ are created in this monitoring module.

---

### Break-Glass Notification Path

Break-glass role usage follows this path:

```text
STS AssumeRole API Call
    |
    v
CloudTrail Event
    |
    v
EventBridge Rule
    |
    v
Security Notifications SNS Topic
    |
    +--> SecOps Email Subscriptions
    +--> Security Notifications SQS Queue
```

If EventBridge cannot deliver the alert to SNS after retries, the failed delivery is retained in:

```text
<name_prefix>-security-notifications-eventbridge-dlq
```

Break-glass usage should be investigated immediately unless it is tied to a known approved emergency.

## Validation

The baseline includes automated validation scripts that should be preferred over one-off manual commands when validating deployed environments.

Recommended validation commands:

```bash
./scripts/validation/validate-sns.sh dev
./scripts/validation/validate-sqs.sh dev
./scripts/validation/validate-eventbridge.sh dev
```

For full environment evidence collection:

```bash
./scripts/validation/validate-all.sh dev
./scripts/validation/export-report.sh dev
```

### Confirm SNS Topics Exist

```bash
aws sns list-topics \
  --region "${AWS_REGION}" \
  --profile "${AWS_PROFILE}" \
  --query 'Topics[?contains(TopicArn, `security-notifications`) || contains(TopicArn, `compliance-notifications`)].TopicArn' \
  --output table
```

Expected:

- Security notifications SNS topic exists
- Compliance notifications SNS topic exists

## Operational Considerations

### Email Subscription Confirmation

SNS email subscriptions do not become active until the recipient confirms the subscription.

After deploying this module, each address in the `secops_emails` variable should receive a confirmation email.

Until confirmed, the subscription remains in:

```text
PendingConfirmation
```

Alerts will not be delivered to that recipient until confirmation is complete.

---

### Durable Notification Queues

The compliance and security notification SQS queues are durable subscribers.

They are useful for:

- Evidence collection
- Troubleshooting
- Replay
- Future SIEM integrations
- Future ticketing integrations
- Client-owned downstream consumers

The baseline does not require active consumers for these queues.

If no consumers are configured, visible messages may accumulate until the retention period expires or the queues are manually drained.

---

### DLQ Review

DLQ messages should be treated as operational evidence and reviewed by SecOps or an authorized operator.

A message in the security notifications SQS DLQ indicates that a message from the security notifications queue failed repeated receive attempts.

A message in the security notifications EventBridge DLQ indicates that EventBridge could not deliver a security notification event to the security notifications SNS topic after applying the configured retry policy.

DLQ messages are not automatically replayed by this module.

---

### CloudTrail Log Dependency

Several detections depend on CloudTrail events being delivered to CloudWatch Logs.

The following detections require a working CloudTrail log group:

- Root activity
- Unauthorized API calls
- CloudTrail disabled events
- IAM policy changes

If CloudTrail is not delivering to the expected log group, these metric filters and alarms will not receive data.

---

### KMS Encryption

SNS topics and SQS queues use the logs CMK.

The logs CMK policy must allow the relevant AWS services to use the key where required.

Services involved may include:

- SNS
- SQS
- CloudWatch
- EventBridge
- AWS Config
- Lambda publishers

If alerts are not being delivered, KMS permissions should be checked along with SNS topic policies and SQS queue policies.

---

### Security Hub Rule Location

The Security Hub high/critical EventBridge rule is not created in this module.

It is expected to be created elsewhere and passed into this module through:

```hcl
securityhub_high_critical_rule_name
securityhub_high_critical_rule_arn
```

This module attaches the security notifications SNS topic as a target and configures the shared security notifications EventBridge DLQ for failed EventBridge-to-SNS delivery.

---

### Tamper Detection Rule Location

The tamper detection EventBridge rule is not created in this module.

It is expected to be created by the security module and passed into this module through:

```hcl
tamper_detection_rule_arn
```

The security notifications SNS topic policy allows that rule to publish alerts.

The security notifications EventBridge DLQ policy allows that rule to send failed target delivery events to the shared EventBridge DLQ.

## Security Notes

- SNS topics are encrypted with the logs CMK.
- Compliance and security notification SQS queues are encrypted with the logs CMK.
- Security notification DLQs are encrypted with the logs CMK.
- SecOps email subscriptions require confirmation.
- Security notifications topic publishing is restricted through topic policy statements.
- EventBridge publish permissions are scoped by source account and source ARN where applicable.
- Compliance queue writes are restricted to the compliance SNS topic.
- Security notification queue writes are restricted to the security notifications SNS topic.
- EventBridge DLQ writes are restricted to approved EventBridge source rule ARNs.
- EventBridge-to-SNS security notification targets use retry policies and DLQs.
- Break-glass role usage generates a critical alert.
- Root user activity generates an alert.
- Unauthorized API calls generate an alert.
- CloudTrail stop/delete activity generates an alert.
- Selected IAM policy changes generate an alert.
- DLQ messages are retained for investigation and are not automatically replayed by this module.

## Design Principles

This module follows:

- Centralized security alerting
- Encrypted notification paths
- Durable notification buffering
- Event-driven monitoring
- Least privilege publishing
- Human-readable security notifications
- Separation of detection and notification routing
- Separation of SNS/SQS redrive DLQs and EventBridge target DLQs
- Compliance event queuing
- Failure-retention paths for security notification delivery
- Fast escalation for critical security events

## Notes

- Deploy this module after logging resources exist.
- The CloudTrail CloudWatch Log Group must exist before metric filters can be attached.
- The logs CMK must allow required AWS services to use encrypted SNS and SQS resources.
- The Security Hub high/critical EventBridge rule is created outside this module.
- The tamper detection EventBridge rule is created outside this module.
- The security notifications SNS topic ARN is consumed by automation and security workflows.
- The compliance SNS topic ARN can be consumed by AWS Config or other compliance routing logic.
- The security notifications EventBridge DLQ ARN is exported for use by other modules that need to protect EventBridge-to-SNS alert delivery.
- For production, confirm all SecOps email subscriptions after deployment.
