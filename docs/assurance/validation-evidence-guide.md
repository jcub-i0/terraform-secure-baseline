# Validation Evidence Guide

## Purpose

This guide explains how to interpret validation evidence generated by the `tf-secure-baseline` validation suite.

The validation suite provides repeatable, read-only checks for deployed workload environments and a separate read-only validation path for the control plane. Its purpose is to help confirm that selected AWS security controls, supporting resources, governance resources, and environment-specific configurations are present and aligned with the expected baseline design.

The validation evidence can support:

- Deployment verification
- Internal security review
- Client handoff
- Troubleshooting
- Lightweight audit-readiness discussions
- Control implementation evidence

---

## Evidence Workflow

The recommended validation evidence workflow is:

1. Deploy or update the target workload environment.
2. Run automated workload validation using `scripts/validation/validate-baseline.sh`.
3. Export workload validation evidence using `scripts/validation/export-baseline.sh`.
4. Run automated control-plane validation using `scripts/validation/validate-control-plane.sh`.
5. Review the generated workload `summary.md`.
6. Retain the generated workload `summary.json` and per-script logs.
7. Retain the control-plane validation output or copied `validate-control-plane.log`.
8. Complete any required manual validation activities.
9. Document warnings, exceptions, and sign-off decisions.

Example workload evidence export command:

```bash
ENV_NAME="dev"

AWS_PROFILE="dev" \
AWS_REGION="us-east-1" \
EXPECTED_ACCOUNT_ID="<account-id>" \
NAME_PREFIX="tf-secure-baseline-${ENV_NAME}" \
./scripts/validation/export-baseline.sh "${ENV_NAME}"
```

Generated workload evidence is written to a timestamped directory under:

```text
validation-results/<environment>/<timestamp>/
```

Example control-plane validation command:

```bash
AWS_PAGER="" \
AWS_PROFILE=control-plane \
AWS_REGION="us-east-1" \
EXPECTED_ACCOUNT_ID="<CONTROL-PLANE-ACCOUNT-ID>" \
EXPECTED_GITHUB_REPOSITORY="<GITHUB-OWNER>/<GITHUB-REPO>" \
ACCOUNT_ID_DEV="<DEV-ACCOUNT-ID>" \
ACCOUNT_ID_STAGING="<STAGING-ACCOUNT-ID>" \
ACCOUNT_ID_PROD="<PROD-ACCOUNT-ID>" \
./scripts/validation/validate-control-plane.sh
```

Generated validation results are ignored by Git by default and should be handled as environment-specific evidence artifacts.

## Report Package Contents

A standard workload validation report package includes:

| File | Purpose |
|---|---|
| `summary.md` | Human-readable workload validation report |
| `summary.json` | Machine-readable workload validation summary |
| `validate-*.log` | Per-script workload validation logs |

The Markdown summary is intended for human review.

The JSON summary is intended for future automation, artifact ingestion, evidence indexing, dashboards, or reporting pipelines.

The per-script logs provide supporting detail for each validation area.

Control-plane validation is currently run separately from workload evidence export. Retain the terminal output or captured log from `validate-control-plane.sh` as supporting evidence for control-plane state backend resources, GitHub OIDC, AWS Organizations, and IAM Identity Center basics.

## What Automated Workload Validation Covers

The automated workload validation suite checks deployed workload environments across the following areas.

| Area | Coverage Summary |
|---|---|
| Environment | Required local tooling, repository paths, Terraform outputs, environment values, and account context |
| Networking | VPC, private subnet routing, NAT Gateway behavior, Network Firewall behavior, and egress-mode expectations |
| VPC Endpoints | Interface endpoint existence, endpoint subnet placement, S3 Gateway Endpoint route table associations, and endpoint availability |
| Logging | CloudTrail, VPC Flow Logs, CloudWatch log groups, metric filters, alarms, retention settings, and log delivery paths |
| Security Services | GuardDuty, Security Hub, AWS Config, Inspector, and profile-aware service expectations |
| KMS | Expected KMS aliases, key state, key manager, and rotation status |
| Backup | Backup vaults, plans, selections, schedules, retention, resources, recent jobs, and recovery point visibility |
| SNS | SNS topics, subscriptions, pending confirmations, and encryption configuration |
| SQS | SQS queues, queue policies, SNS delivery paths, encryption, dead-letter queue status, and message counts |
| EventBridge | Default-bus and SecOps-bus rules, targets, rule state, target DLQs, retry policies, and rollback rule coverage |
| Lambda | Lambda functions, runtime, execution role, timeout, memory, KMS configuration, VPC configuration, environment variables, and invoke permissions |
| SSM | Managed instance registration, online status, associations, maintenance windows, and patch baseline visibility |
| Compute | EC2 instance placement, public IP absence, IMDSv2 enforcement, detailed monitoring, instance profiles, security groups, required tags, isolation eligibility, and EBS encryption |
| IAM | IAM roles, service trust policies, GitHub OIDC roles where present, break-glass MFA conditions, and shared log access policies |

## What Automated Control-Plane Validation Covers

Control-plane validation is handled separately from workload validation because the control plane manages bootstrap and governance resources rather than workload baseline infrastructure.

| Area | Coverage Summary |
|---|---|
| AWS Identity | Confirms the active AWS caller identity and optional expected control-plane account ID |
| Control-Plane State Outputs | Confirms required Terraform state stack outputs are present where applicable |
| State S3 Bucket | Confirms the Terraform state bucket exists, has versioning, public access block settings, and SSE-KMS encryption |
| State KMS CMK | Confirms the Terraform state KMS key exists, is enabled, and is customer-managed |
| State Lock Table | Confirms the DynamoDB state lock table exists and is active |
| GitHub OIDC | Confirms the GitHub OIDC provider exists |
| GitHub Plan/Apply Roles | Confirms control-plane GitHub plan/apply roles exist |
| GitHub Trust Conditions | Confirms role trust policies reference the expected GitHub repository where `EXPECTED_GITHUB_REPOSITORY` is provided |
| AWS Organizations | Confirms Organizations is accessible and expected OU structure exists |
| Account Placement | Optionally checks workload account placement under expected OUs, generally warning unless strict mode is enabled |
| IAM Identity Center | Confirms Identity Center instance discovery, expected SecOps groups, permission sets, and account assignments |

A successful control-plane validation run should end with:

```text
[PASS] Control-plane validation completed successfully
```

## What Automated Validation Does Not Cover

The automated validation scripts are intentionally read-only.

They do not perform destructive, privileged, live-response, or state-changing tests.

The validation suite does not perform:

- Live EC2 isolation
- Live EC2 rollback
- Live IP enrichment execution
- Tamper detection simulation
- Break-glass role assumption
- End-user IAM Identity Center login testing
- Identity Center group membership approval or access review
- Identity Center assignment changes
- GitHub Actions workflow execution
- Destroy workflow execution
- AWS account movement between Organizations OUs
- Manual console review
- Control owner interviews
- Policy review
- Evidence sampling
- Risk assessment
- Formal audit testing

These activities may be required depending on client expectations, audit scope, risk profile, and environment maturity.

## Manual Tests

The following validation areas remain manual by design.

| Manual Test | Purpose |
|---|---|
| GitHub Actions workflow validation | Confirms CI/CD workflows run successfully with the intended OIDC roles and environment protections |
| IAM Identity Center end-user login and effective access testing | Confirms users can sign in through IAM Identity Center and receive the intended effective access |
| Identity Center group membership review | Confirms users, groups, permission sets, and account assignments match the intended access model |
| Live EC2 isolation test | Confirms the isolation Lambda can quarantine an instance during a controlled test |
| Live EC2 rollback test | Confirms the rollback workflow can restore original security groups after approved rollback |
| Live IP enrichment test | Confirms Security Hub findings containing public IPs trigger enrichment behavior |
| Tamper detection test | Confirms attempts to disable or alter security services generate expected detections and notifications |
| Break-glass role assumption test | Confirms emergency access behaves as expected and requires MFA where configured |
| Destroy safety review | Confirms teardown workflows are understood, controlled, and safe for the target environment |

## Interpreting PASS, WARN, and FAIL

### PASS

A `PASS` result means the validation script completed successfully and confirmed the expected condition for that environment or control-plane scope.

A passing result should be interpreted in the context of the selected deployment profile, effective Terraform outputs, and whether the check applies to workload infrastructure or control-plane governance resources.

For example, a development environment may pass even if AWS Backup is disabled, as long as the effective profile settings indicate that backups are not required for that environment.

A control-plane validation run may pass while still producing non-blocking governance warnings, such as account OU placement warnings, if those checks are intentionally warning-only.

### WARN

A `WARN` message indicates something that may require review but does not necessarily mean the environment is invalid.

Warnings may be expected in some environments.

Examples include:

- Optional resources not present
- Pending SNS email confirmation
- No custom patch baseline configured
- Backup resources skipped because backups are disabled
- Network Firewall absent because the environment uses `nat_only`
- NAT Gateway absent because the environment uses `vpc_endpoints_only`
- Workload accounts located under the AWS Organizations root when Terraform currently creates OU structure but does not manage account placement

Warnings should be reviewed before client handoff or sign-off.

### FAIL

A `FAIL` result means a required validation check did not pass.

Failures should be reviewed before relying on the environment or control plane as validated evidence.

A failed validation script may indicate:

- Missing Terraform outputs
- Missing AWS resources
- Incorrect account or region
- Incorrect `NAME_PREFIX`
- Incorrect deployment profile assumptions
- Incorrect control-plane account profile
- Incorrect expected GitHub repository value
- IAM trust policy issues
- Failed AWS CLI calls
- Missing required local tooling
- Mismatched resource naming
- Incomplete deployment

---

## Control-Plane Warning Behavior

Control-plane validation may intentionally treat some governance findings as warnings rather than failures.

For example, AWS Organizations OU structure may be validated successfully while workload account placement produces warnings. This is acceptable when the Organizations stack creates the `Workloads`, `NonProd`, and `Prod` OUs but does not manage AWS account creation, invitations, or movement between OUs.

Treat control-plane warnings as follow-up items unless the engagement, client policy, or Terraform design makes the condition mandatory.

If account placement becomes a strict requirement, document that requirement and enable stricter validation behavior where supported.

## Compliance Positioning

The validation suite can support audit-readiness and control implementation discussions, but it should not be represented as a certification or audit opinion.

The validation suite confirms the presence and configuration of selected AWS security controls, governance resources, and supporting infrastructure at the time validation was performed.

The validation suite does not replace:

- A full compliance program
- A SOC 2 audit
- An ISO 27001 audit
- Internal control design review
- Control operating effectiveness testing
- Formal audit evidence review
- Policy and procedure documentation
- Risk assessment
- ISMS implementation
- Management review
- Vendor risk management
- Legal or regulatory review

Recommended positioning:

- “This report provides deployment validation evidence.”
- “This report supports audit-readiness efforts.”
- “This report confirms selected AWS security controls were present and configured at the time of validation.”
- “This report should be reviewed alongside policies, procedures, risk assessments, and other compliance evidence.”

Avoid positioning such as:

- “This environment is SOC 2 compliant.”
- “This environment is ISO 27001 certified.”
- “This report proves compliance.”
- “This report replaces an audit.”
- “This report proves operating effectiveness.”

## Evidence Retention

Validation evidence should be retained according to the organization’s evidence retention expectations.

Recommended retention considerations:

| Evidence Type | Recommended Handling |
|---|---|
| `summary.md` | Retain with deployment or client handoff records |
| `summary.json` | Retain for automation, indexing, dashboarding, or evidence package generation |
| Per-script workload logs | Retain as supporting technical evidence |
| `validate-control-plane.log` or captured control-plane output | Retain as supporting control-plane technical evidence |
| Manual validation notes | Retain with sign-off or engagement records |
| Exceptions | Retain with risk acceptance or remediation tracking |

Generated evidence may contain:

- AWS account IDs
- AWS regions
- Resource names
- ARNs
- Environment names
- Validation timestamps
- Operational configuration details
- GitHub organization or repository names
- IAM Identity Center group and permission set names

Treat validation evidence as internal or client-confidential unless explicitly approved for public release.

## Recommended Client Handoff Process

For a client-facing handoff, provide:

1. Workload `summary.md`
2. Workload `summary.json`
3. Per-script workload validation logs
4. Control-plane validation output or `validate-control-plane.log`
5. Completed manual validation notes, where applicable
6. Documented warnings or exceptions
7. Limitations language
8. Sign-off section or acceptance record

Recommended handoff message:

> The attached validation package summarizes automated read-only validation results for the deployed workload environment and supporting control-plane resources. The package includes a human-readable workload summary, a machine-readable workload summary, per-script workload logs, and control-plane validation evidence covering state backend resources, GitHub OIDC, AWS Organizations OU structure, and IAM Identity Center basics. Manual validation items and limitations are documented separately and should be reviewed before relying on the environment for production or audit-readiness purposes.

---

## Recommended Internal Review Process

Before using validation evidence for client delivery or internal sign-off:

1. Confirm the correct AWS account and region were validated.
2. Confirm the correct workload environment was validated.
3. Confirm `EXPECTED_ACCOUNT_ID` matched the active AWS caller identity.
4. Confirm the `NAME_PREFIX` matches deployed workload resource names.
5. Confirm control-plane validation was run with the control-plane profile.
6. Confirm `EXPECTED_GITHUB_REPOSITORY` matched the repository trusted by the control-plane GitHub OIDC roles.
7. Review all failed checks.
8. Review all warnings.
9. Confirm manual validation items are tracked.
10. Confirm exceptions are documented.
11. Confirm evidence files are stored in the appropriate location.
12. Confirm generated evidence is not committed to the source repository.

---

## Limitations

This guide explains how to interpret validation evidence generated by the `tf-secure-baseline` validation suite.

It does not define a complete compliance program, risk management process, audit methodology, or operating model.

Validation results should be reviewed by qualified technical and security personnel before being used for client handoff, audit-readiness discussions, or production acceptance.
